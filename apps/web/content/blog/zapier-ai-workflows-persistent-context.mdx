Zapier added AI features rapidly over the past year. You can now use OpenAI, Claude, and custom AI actions directly in your Zaps. But one thing hasn't changed: **Zapier Zaps are stateless.**

Every Zap trigger is a fresh start. The AI step doesn't know what happened last time this Zap ran, which users it's spoken to, or what was decided three runs ago.

For simple tasks — "summarize this email," "extract data from this PDF" — statelessness is fine. But for AI agents that handle ongoing relationships, it's a dealbreaker.

This guide shows you how to fix it using retainr's REST API in Zapier's built-in HTTP action.

## Why Zapier Users Hit This Wall

Zapier's AI steps are designed for single-step, single-run processing. The product metaphor is "trigger → action," not "trigger → reason → remember → act."

Common scenarios where this breaks:

**Lead nurturing bot**: Zap fires when a lead submits a form. AI generates a personalized follow-up. But the AI has no idea this lead submitted a form three weeks ago and already got the first follow-up.

**Customer support routing**: AI classifies incoming support tickets. But it can't learn that this particular customer always has billing-related issues and should go straight to the billing team.

**Content personalization**: AI generates content for a newsletter subscriber. But it doesn't know the subscriber's past preferences, clicks, or feedback.

In all these cases, you need memory.

## Zapier + retainr: The Pattern

Since Zapier doesn't have a native retainr app yet (it's coming), you use Zapier's **Webhooks by Zapier** or **Code by Zapier** actions to call retainr's REST API directly.

The API is simple: two calls per Zap run — search before the AI step, store after.

### Step 1: Search for relevant memories

Add a **Webhooks by Zapier** step with:

- **Method**: GET
- **URL**: `https://api.retainr.dev/v1/memories/search`
- **Headers**:
  - `Authorization: Bearer YOUR_RETAINR_API_KEY`
  - `Content-Type: application/json`
- **Data** (as query params):
  - `query`: The incoming message or context to search against
  - `user_id`: The user's email, phone, or unique ID
  - `limit`: 5

The response is a JSON array of relevant past memories, sorted by semantic similarity.

### Step 2: Format memories for your AI prompt

Add a **Formatter by Zapier** step to join the memory contents:

Use the "Text" formatter with the "Join" transform. Select the content fields from your webhook response.

Or, in your AI step's prompt, reference the array directly:

```
You are a helpful assistant.

Past interactions with this user:
{{format_step.output}}

Current message: {{trigger.message}}
```

### Step 3: Run your AI step

Now your OpenAI or Anthropic step receives context from past interactions. The AI can reference previous conversations, preferences, and decisions.

### Step 4: Store the interaction

After the AI step, add another **Webhooks by Zapier** action:

- **Method**: POST
- **URL**: `https://api.retainr.dev/v1/memories`
- **Headers**: `Authorization: Bearer YOUR_RETAINR_API_KEY`
- **Body** (JSON):

```json
{
  "content": "User message: [the input]\nAI response: [the AI output]",
  "user_id": "[user email or ID]",
  "tags": ["zapier", "conversation"]
}
```

That's the complete pattern. Your Zap now has persistent memory.

<Callout type="tip">
Store both the user's message AND the AI response together. When you search later, the stored content helps find contextually relevant memories even if the exact wording differs.
</Callout>

## Complete Example: Lead Follow-Up Zap

Here's a complete Zap for a lead nurturing sequence:

**Trigger**: New form submission (Typeform, Gravity Forms, whatever)

**Step 1 — Webhooks: Search Memory**
```
GET https://api.retainr.dev/v1/memories/search
?query={{lead.email}} lead history
&user_id={{lead.email}}
&limit=5
```

**Step 2 — OpenAI: Generate follow-up**
System prompt:
```
You are a sales assistant for [Company]. Write a personalized follow-up email.

History with this lead:
{{step1.results[*].content | join("\n---\n")}}

Lead info:
- Name: {{lead.name}}
- Company: {{lead.company}}
- Interest: {{lead.interest}}

Write a warm, specific follow-up. Reference past interactions if relevant.
Don't be generic. 150 words max.
```

**Step 3 — Send email** (Gmail, SendGrid, whatever)

**Step 4 — Webhooks: Store Memory**
```json
{
  "content": "Lead {{lead.name}} from {{lead.company}} submitted form about {{lead.interest}}. Sent follow-up: {{step2.choices[0].message.content}}",
  "user_id": "{{lead.email}}",
  "tags": ["lead", "email", "follow-up"]
}
```

Now when this lead submits again, or when a sales rep asks the AI about them, the full history is available.

## Advanced: Tag-Based Memory Categories

For complex Zaps, organize memories with tags so you can search within categories:

```json
{
  "content": "Customer escalated billing dispute for invoice #4521. Refund approved for €150.",
  "user_id": "customer@example.com",
  "tags": ["billing", "escalation", "refund"]
}
```

Then when handling a new billing issue, search with tag filter:
```
GET /v1/memories/search?query=billing+dispute&user_id=customer@example.com&tags=billing
```

This returns only billing-related memories, avoiding noise from other interaction types.

## Zapier-Specific Considerations

**Multi-step Zap cost**: Each additional step increases your Zap's operation count. Two extra steps per run (search + store) affects your Zapier plan limits. Factor this into your plan selection.

**Rate limiting**: retainr's free tier allows 1,000 operations/month. At 2 ops per Zap run, that's 500 Zap runs. Builder tier (€29/month) covers 10,000 Zap runs per month.

**Error handling**: Add Zapier's built-in error handling to your webhook steps. If retainr is unavailable, your Zap should continue without memory rather than failing entirely.

**Testing**: Zapier makes it easy to test each step individually. Test your search step first to verify you get results back, then test the full Zap.

## When Zapier vs. When n8n/Make.com

Zapier's HTTP approach works well for occasional Zaps with memory. For high-volume AI workflows where memory is central to the product, n8n or Make.com are better choices because:

- They have native retainr modules (no manual HTTP setup)
- More flexible data transformation
- Lower per-operation cost at scale

Zapier shines when you're connecting many different apps quickly and memory is one component among many.
