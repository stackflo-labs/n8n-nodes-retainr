Customer service AI is everywhere now. Most of it is frustrating — generic responses, no context, "I don't understand your question" loops. The reason is almost always the same: **the AI doesn't know who the customer is or what happened before.**

This tutorial builds a customer service AI that actually works. It remembers customers, personalizes responses, and gets smarter over time — built entirely in Make.com, no code required.

## What We're Building

A customer service AI that:
- Responds to incoming support requests (email, chat, form — your choice)
- Searches past interactions before responding
- Generates personalized, context-aware responses
- Classifies tickets (billing, technical, general) and routes appropriately
- Learns from every interaction, improving over time
- Escalates to humans when confidence is low

We'll use:
- **Make.com** for the automation logic
- **OpenAI GPT-4** for the AI responses
- **retainr** for persistent customer memory
- **Gmail** for email handling (swap for Intercom, Freshdesk, etc.)

## Part 1: Setting Up Your Tools

### Get a retainr API key

Make.com has a built-in Gmail connector. Connect it with OAuth — standard flow.

## Part 2: Building the Core Scenario

### Step 1: Email trigger

Add a **Gmail: Watch Emails** trigger. Configure:
- Label: `customer-support` (create this label in Gmail and have customers email support@yourdomain.com, which auto-labels)
- Mark as read: Yes

This fires every time a new support email arrives.

### Step 2: Extract customer info

Add a **Text Parser** module to extract the customer's email address from the message:

```
Pattern: From: .*<(.+)>
Text: {{1.from}}
```

Or use the email field directly from the trigger: `{{1.from.email}}`

### Step 3: Search customer memory

Add a **retainr: Search Memory** module:

| Field | Value |
|-------|-------|
| Query | `{{1.subject}} {{1.snippet}}` |
| User ID | `{{1.from.email}}` |
| Limit | 5 |

This finds the 5 most relevant past interactions for this customer based on their current message.

### Step 4: Classify the ticket

Add an **OpenAI: Create a Completion** module for classification:

System prompt:
```
Classify this customer support request into exactly one category: BILLING, TECHNICAL, ACCOUNT, GENERAL.
Respond with only the category name.
```

User message: `{{1.subject}}: {{1.text}}`

### Step 5: Generate the response

Add another **OpenAI: Create a Completion** module for the actual response:

System prompt:
```
You are a helpful customer service agent for [Your Company].

CUSTOMER HISTORY (most relevant past interactions):
{{3.memories[].content | join("\n\n---\n\n")}}

GUIDELINES:
- Reference relevant past interactions naturally when appropriate
- Don't repeat information they've already been told
- If this is a billing issue, always include a reference to our billing team email
- Be warm and specific, not generic
- Keep response under 200 words unless complexity requires more
- If you're unsure, say so and offer to escalate

TICKET TYPE: {{4.result}}
```

User message:
```
From: {{1.from.email}}
Subject: {{1.subject}}

{{1.text}}
```

### Step 6: Route based on ticket type

Add a **Router** module with three paths:

**Path 1 — BILLING**: Forward to billing@yourcompany.com with the AI draft
**Path 2 — TECHNICAL**: Assign to technical queue (or reply directly if simple)
**Path 3 — GENERAL**: Send the AI response directly

For this tutorial, let's assume General tickets get auto-replied. Add a **Gmail: Send Email** module on that path.

### Step 7: Store the interaction in memory

**Critical step** — add a **retainr: Store Memory** module after the response is sent:

| Field | Value |
|-------|-------|
| Content | `Customer inquiry [{{4.result}}]: {{1.subject}}. AI response: {{5.choices[].message.content}}` |
| User ID | `{{1.from.email}}` |
| Tags | `support, {{4.result}}, {{formatDate(now, "YYYY-MM")}}` |

The month tag lets you later search "what issues did this customer have in March 2026?"

## Part 3: Handling Edge Cases

### When the AI is uncertain

Add a confidence check. Modify your response prompt to include:

```
End your response with exactly one line: CONFIDENCE: HIGH, MEDIUM, or LOW
```

Then add a **Text Parser** to extract the confidence level, and route LOW confidence tickets to a human review queue instead of auto-sending.

### Preventing repeat answers

Include a check in your search step. If the customer asked the same question before:

```
IMPORTANT: If the customer has asked this before (visible in their history),
acknowledge that they've encountered this issue again and try a different solution approach.
```

### Handling angry customers

Add an **OpenAI: Create a Completion** step to detect sentiment:

```
Classify the sentiment of this message: POSITIVE, NEUTRAL, FRUSTRATED, ANGRY
Message: {{1.text}}
```

Route ANGRY sentiment tickets directly to your human team, no AI response.

## Part 4: Making It Smarter Over Time

The system improves automatically because every stored memory informs future responses. But you can accelerate this:

**Store resolution outcomes**: When a ticket is closed, add a Make.com scenario that stores:
```
Issue resolved: [description]. Solution that worked: [what fixed it].
Time to resolve: [duration].
```

**Store customer preferences**: When a customer mentions a preference ("please only contact me by email"), store it:
```
Customer preference: email-only communication. Set on {{date}}.
```

**Tag sentiment**: Tag memories with sentiment so you can search "what issues make customers angry" and proactively fix them.
