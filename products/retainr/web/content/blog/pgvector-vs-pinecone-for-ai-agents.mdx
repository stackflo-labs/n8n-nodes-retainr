The vector database market exploded over the past two years. Pinecone, Weaviate, Qdrant, Chroma, Milvus — there are more options than most teams can evaluate.

For AI agent memory specifically, the comparison that comes up most often is **pgvector vs Pinecone**. One is a Postgres extension. The other is a purpose-built vector database with a $100M+ funding pedigree.

This post cuts through the marketing and answers the actual question: which one should you use for AI agent memory, at what scale, and what does it cost?

## What We're Actually Comparing

Before the comparison, a clarification: pgvector is a Postgres *extension*, not a standalone product. You run it on a Postgres instance — self-hosted, on RDS, on Supabase, or on Neon. It adds vector column types and similarity search operators to standard SQL.

Pinecone is a managed vector database service. You call their API to upsert vectors and query them. There's no SQL, no schemas, no joins.

This is a meaningful distinction because the choice isn't just about vector search performance — it's about your entire data architecture.

## Performance: The Real Numbers

Let's start with what actually matters for AI agent memory workloads.

### Query Latency

Typical workload: a user sends a message, you search their memory for relevant context, you inject it into the prompt. The memory search needs to complete in under 200ms to not meaningfully impact response time.

**pgvector (HNSW index, self-hosted on 4-core machine)**:
- 10,000 vectors: ~5-10ms
- 100,000 vectors: ~20-50ms
- 1,000,000 vectors: ~50-120ms

**Pinecone (s1 pod, standard tier)**:
- 10,000 vectors: ~30-60ms (network included)
- 100,000 vectors: ~30-80ms
- 1,000,000 vectors: ~40-100ms

At the scales relevant to most AI agent deployments (under 1M vectors), pgvector is as fast as Pinecone — sometimes faster, because there's no network hop to an external service.

<Callout type="tip">
pgvector's HNSW index is the key to performance. Make sure you create it explicitly: `CREATE INDEX ON memories USING hnsw (embedding vector_cosine_ops)`. Without it, pgvector falls back to a full table scan, which is slow at any scale.
</Callout>

### Write Throughput

For agent memory, writes happen after each interaction. This is typically 1-10 writes per second for most deployments.

Both pgvector and Pinecone handle this trivially. Write throughput only becomes a concern above 1,000 writes/second — a scale that applies to very few AI agent deployments.

### Cold Start

Pinecone has no cold start. pgvector is subject to the normal Postgres startup behavior — which on managed services like RDS or Supabase is a non-issue since the instance is always running.

## Cost Analysis

This is where the comparison gets interesting.

### For Small Deployments (under 100k vectors)

**pgvector** (on a $10/month Hetzner VPS or shared RDS):
- Compute: $10-30/month
- Storage: negligible
- Total: **$10-30/month**

**Pinecone** (free tier: 100k vectors, 1 namespace):
- Total: **$0/month**

Pinecone wins for small deployments. Their free tier is genuinely generous and covers most early-stage AI products.

### For Medium Deployments (100k - 10M vectors)

**pgvector** (on a 4-core/8GB VPS or small RDS):
- Compute + storage: $50-150/month
- Total: **$50-150/month**

**Pinecone** (s1 pods, serverless, growing with usage):
- Serverless: ~$0.04/1M read units + $2/GB storage
- 10M vectors at 1536 dims = ~24GB → $48/month storage
- Plus read operations: at 100k searches/month = $4
- Total: **~$52/month**

Cost parity at this range. The difference is operational complexity.

### For Large Deployments (10M+ vectors)

**pgvector** (dedicated 8-core/32GB Postgres instance):
- Compute: $200-500/month
- Storage: $50-100/month
- Total: **$250-600/month**

**Pinecone** (serverless, heavy usage):
- Storage: $480/month (240GB for 100M vectors)
- Read operations: 1M searches/month = $40
- Total: **$520+/month**

At large scale, costs are roughly comparable, but pgvector gives you more control over optimization.

<Callout type="warning">
These are rough estimates. Pinecone pricing depends heavily on your dimension count, operation frequency, and replication settings. Always run your own estimate at [pinecone.io/pricing](https://www.pinecone.io/pricing) with your actual numbers.
</Callout>

## Complexity Comparison

### pgvector

**What you manage**:
- Postgres instance (backups, scaling, updates)
- HNSW index tuning (m, ef_construction parameters)
- Schema design (vectors live alongside your other data)
- Query optimization (EXPLAIN ANALYZE on slow queries)

**What you get**:
- SQL joins between vectors and your regular data
- ACID transactions (vector upsert + metadata update in one transaction)
- Row-level security (filter vectors by userId with RLS policies)
- No additional API to integrate — same database your app already uses

### Pinecone

**What you manage**:
- Pinecone API integration
- Metadata sync (your business data lives elsewhere; you duplicate relevant metadata into Pinecone)
- Namespace management (equivalent to user isolation)

**What you get**:
- Zero infrastructure management
- Simple upsert/query API
- Managed scaling
- Dedicated vector operations UI

The pgvector complexity is real but front-loaded. Once your schema and indexes are set up, it runs on autopilot. The Pinecone simplicity is real but comes with a hidden cost: **data fragmentation**.

## The Data Fragmentation Problem with Pinecone

This is the argument for pgvector that doesn't get talked about enough.

Your AI agent memory exists alongside other data: user records, subscription status, conversation metadata, timestamps, tags. With pgvector, all of this lives in one database. A single SQL query can filter memories by user, join to the subscription table to check their plan, and return only memories from the last 30 days.

```sql
SELECT m.content, m.created_at, m.tags
FROM memories m
JOIN users u ON m.user_id = u.id
WHERE m.user_id = 'user_123'
  AND u.plan = 'pro'
  AND m.created_at > NOW() - INTERVAL '30 days'
ORDER BY m.embedding <=> query_embedding
LIMIT 5;
```

With Pinecone, you'd need to:
1. Query Pinecone for similar vectors
2. Filter by userId in Pinecone metadata
3. Take the returned IDs
4. Query your main database for full record details
5. Join the results yourself

Every query becomes a multi-step operation across two systems. Consistency becomes a concern. If you update a user's subscription in your main database, you need to remember to also update their Pinecone metadata. These get out of sync.

<Callout type="tip">
If your AI agent needs to filter memories by user attributes (plan, signup date, region), pgvector is significantly simpler. Pinecone's metadata filtering is limited compared to SQL WHERE clauses.
</Callout>

## When to Actually Use Pinecone

pgvector is better for most AI agent workloads, but Pinecone is genuinely the right choice in specific situations:

**Use Pinecone when:**
- You need to store 100M+ vectors and don't want to manage Postgres scaling
- Your team has zero database operations experience and needs zero-ops infrastructure
- You need multi-region replication for vectors specifically
- You're already in a serverless architecture and want a consistent operational model
- You're early-stage and the free tier covers your current scale

**Use pgvector when:**
- Your vectors need to join with relational data (almost always true for user memory)
- You already run Postgres (adding pgvector is just `CREATE EXTENSION`)
- You want ACID transactions spanning vector and non-vector data
- You need row-level security (user isolation at the database layer)
- You want to keep your infrastructure simple (one database, not two)

## Why retainr Uses pgvector

We built retainr on pgvector for three reasons:

**1. User isolation via RLS.** Each workspace's memories are scoped with PostgreSQL row-level security policies. A bug in the API layer cannot leak User A's memories to User B — the database enforces the boundary. This is a hard requirement for a multi-tenant memory product.

**2. Transactional integrity.** When you store a memory and update your usage counter in the same request, it happens atomically. Either both succeed or neither does. With Pinecone + a separate database, you'd need distributed transactions to achieve this.

**3. SQL flexibility.** We can run complex analytics queries on memories — who's storing the most, which tags appear together, what percentage of memories are being recalled. This is native SQL. With Pinecone, we'd need a separate analytics system.

The tradeoff: we have to manage Postgres infrastructure. For retainr's scale and requirements, that's a reasonable trade.

## Benchmark: pgvector HNSW vs. Flat Scan

If you're running pgvector yourself, this is the most important configuration decision.

Without HNSW index:
```sql
-- Full table scan — O(n), slow at scale
SELECT content FROM memories
ORDER BY embedding <=> query_embedding
LIMIT 5;
-- 100k rows: ~800ms
```

With HNSW index:
```sql
CREATE INDEX ON memories USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);
-- Same query
-- 100k rows: ~20ms
```

The HNSW index reduces query time by 40-50x. It uses approximate nearest-neighbor search, which means results are not perfectly optimal — but for AI agent memory retrieval, the approximate results are indistinguishable from exact results in practice.

Key HNSW parameters:
- `m = 16` — number of connections per node. Higher = better recall, more memory.
- `ef_construction = 64` — build-time accuracy. Higher = slower builds, better index quality.
- Default values work well for most agent memory workloads up to 10M vectors.
