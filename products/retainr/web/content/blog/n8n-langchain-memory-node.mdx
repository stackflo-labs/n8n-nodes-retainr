If you've built an AI agent in n8n, you've probably tried the built-in memory nodes. They look like the right solution. They're called "memory." They're right there in the LangChain node group.

And then you trigger your workflow a second time and realize: **it forgot everything.**

This post explains exactly why that happens, what the n8n memory nodes actually do, and how to replace them with persistent memory that survives across workflow executions.

## What n8n's LangChain Memory Nodes Actually Do

n8n ships with several memory nodes in the AI category:

- **Window Buffer Memory** — keeps the last N messages in a sliding window
- **Token Buffer Memory** — keeps messages up to a token limit
- **Simple Memory** — basic key-value store for conversation context

These nodes are wrappers around LangChain's memory abstractions. They're useful in a specific context: **within a single workflow execution**.

If you're building a multi-step agent that calls tools, loops, and needs to remember earlier steps in the *same run*, these nodes work fine. The conversation history is maintained in memory as the execution progresses.

The problem is what happens when the execution ends.

## Why Memory Resets Between Executions

n8n is a workflow orchestrator, not a stateful application server. When a workflow execution completes:

1. All in-memory data is freed
2. The execution record is written to the database (just logs and metadata)
3. No business data from that run persists anywhere

The LangChain memory nodes store their data in the execution's memory space. When the execution ends, that memory is gone.

<Callout type="warning">
This applies even to n8n's "Chat Memory Manager" nodes. The memory they manage is scoped to the execution. There is no built-in persistence layer that survives across separate webhook calls or scheduled runs.
</Callout>

Here's the execution model visualized:

```
Execution 1: [User: "Hi"] → [Memory stored in RAM] → Execution ends → RAM freed
Execution 2: [User: "Remember me?"] → Fresh start → Memory = empty
```

There's no bridge between executions. Each webhook call or trigger starts a new, isolated execution.

## The Window Buffer Memory Misconception

The Window Buffer Memory node has a `sessionId` field, which misleads many people into thinking it provides cross-execution persistence.

It doesn't.

The `sessionId` is used within an execution to partition memory if you're running multiple agent chains in the same workflow. It's a namespace, not a persistence key.

When you set `sessionId` to the user's ID and then trigger the workflow again, the new execution has no access to the previous execution's Window Buffer Memory. The session is gone.

## What "Persistent" n8n Memory Actually Requires

To have memory that genuinely persists across executions, you need to store it somewhere *outside* of the execution's memory space. Options:

| Storage | Queryable? | Vector Search? | Setup Effort |
|---------|-----------|---------------|-------------|
| n8n Static Data | No | No | Low (but fragile) |
| Google Sheets | Sort of | No | Medium |
| Airtable | Yes (limited) | No | Medium |
| Postgres (external) | Yes | With pgvector | High |
| Redis | Yes | With modules | High |
| retainr | Yes | Yes (built-in) | Low |

The fundamental issue with n8n's built-in memory is that it's designed for single-execution context management, not multi-execution persistence. These are different problems.

## The Real Cost of Fake Memory

Let's look at what happens when you ship an AI agent without real persistent memory:

**Support bot scenario**: User contacts support, explains they're on the Pro plan, mentions they have three team members, describes their use case in detail. The AI responds helpfully. Next week, the user asks a follow-up question. The bot asks: "What plan are you on?"

The user experience is broken. The bot appears dumb. Users lose trust.

**Sales assistant scenario**: Prospect has three conversations about pricing. On conversation four, the AI starts the pricing explanation from scratch. The prospect concludes the company's AI is a basic FAQ bot, not a real assistant.

**Project management bot**: User is tracking a long-running project. After a weekend, the bot has no memory of the project status, decisions made, or open questions. Everything has to be re-explained.

In each case, the agent is worse than useless — it creates the *illusion* of being smart while behaving in ways that feel random and frustrating.

## Setting Up Real Persistent Memory in n8n

Here's how to replace n8n's built-in memory with persistent, cross-execution memory using retainr.

### Step 1: Remove the LangChain Memory Nodes

If you have Window Buffer Memory or Simple Memory nodes in your workflow, remove them. They'll cause confusion once you have real memory working.

### Step 2: Install the retainr Community Node

In n8n, go to **Settings → Community Nodes → Install**.

Search for `n8n-nodes-retainr` and install it. Restart n8n if required.

Create a **retainr API** credential with your API key from [retainr.dev/dashboard](/dashboard).

### Step 3: Add Memory Retrieval Before the AI Node

At the start of your workflow, before any AI node, add a **retainr → Search Memory** node:

```json
{
  "query": "{{ $json.chatInput }}",
  "userId": "{{ $json.sessionId }}",
  "limit": 5
}
```

Use `sessionId` from the n8n webhook input as your userId. This is the stable identifier n8n uses to track conversations.

### Step 4: Format the Memories for the AI

Add a **Set** node to format the memory results:

**Field name**: `memoryContext`

**Field value** (JavaScript expression):

```javascript
{{ $json.memories.length > 0
  ? "Previous conversation context:\n\n" + $json.memories.map(m => m.content).join('\n\n')
  : "" }}
```

### Step 5: Update Your AI Agent Node

In the AI Agent node's system prompt, inject the memory context:

```
You are a helpful assistant.

{{ $('Format Memories').item.json.memoryContext }}

Answer the user's current question based on this context. If you have relevant history, reference it naturally. If not, respond normally.
```

### Step 6: Store the Interaction After Response

Add a **retainr → Store Memory** node after the AI Agent:

```json
{
  "content": "User: {{ $('Webhook').item.json.chatInput }}\n\nAssistant: {{ $json.output }}",
  "userId": "{{ $('Webhook').item.json.sessionId }}",
  "tags": ["conversation"]
}
```

This is the complete replacement for LangChain memory. It's six nodes total, all visual, no code required.

## Migration Guide: Converting Existing n8n Agents

If you have a working (but amnesiac) n8n AI agent and want to add persistent memory, here's the migration path:

**1. Map your existing execution flow**

Identify: where does user input enter? Where does the AI respond? What's the userId/sessionId?

**2. Keep your LangChain nodes for intra-execution context**

If you have a complex agent that calls multiple tools in one execution, keep the Window Buffer Memory for that within-execution context. It still works for that purpose.

**3. Add retainr as the inter-execution layer**

At the start: search for past context. At the end: store the current interaction. The LangChain memory handles the current run; retainr handles history.

**4. Gradually phase out LangChain memory**

Once retainr is working, you can often simplify your workflow by removing the LangChain memory nodes entirely. The vector search retrieves what's relevant, so you don't need to maintain a sliding window of recent messages.

<Callout type="tip">
During the migration, keep both systems running in parallel for a week. Compare the AI responses. Once you're confident retainr's retrieval is producing better context than the sliding window, remove the LangChain memory nodes.
</Callout>

## Common Questions

**Does this work with n8n Cloud?** Yes. The retainr community node is compatible with n8n Cloud v1.0+. Install it via Settings → Community Nodes.

**What's the difference between sessionId and userId in retainr?** `userId` scopes memories permanently to a user. `sessionId` scopes memories to a specific session within a user's history. Use userId for customer memory, sessionId for project-specific or conversation-thread-specific memory.

**Can I import existing n8n conversation logs into retainr?** Yes — use the retainr API's bulk-store endpoint. Export your n8n execution history, format as memory objects, and POST them in batches. This is useful for seeding memory from historical data.

**What if I want to clear memory for a specific user?** Call `DELETE /v1/memories` with the `userId` parameter. All memories for that user are removed immediately. This is important for GDPR right-to-erasure requests.
